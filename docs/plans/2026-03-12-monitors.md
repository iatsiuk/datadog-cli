# Monitors -- Core Alerting

## Overview
- Implement full monitors management: CRUD, search, mute/unmute
- Implement downtimes management: CRUD (create/cancel scheduled downtimes)
- Implement monitor config policies: CRUD
- Enables CLI-based alerting workflow without Datadog UI

## Context
- Existing patterns: `internal/cmd/events.go`, `internal/cmd/logs.go` -- API wrapper struct, DI via `mkAPI`, cobra subcommands
- APIs: `datadogV1.MonitorsApi`, `datadogV2.DowntimesApi`, `datadogV2.MonitorsApi` (config policies)
- Output: `internal/output` package (PrintTable/PrintJSON)
- Tests: httptest mock server pattern, table-driven tests
- Registration: `internal/cmd/root.go` -- `NewMonitorsCommand()` added to root

## Technical Details
- **Monitor types**: metric, query alert, composite, service check, event alert, etc.
- **Thresholds**: critical, warning, ok, critical_recovery, warning_recovery -- pass as JSON string flag
- **Downtime schedule**: one-time (start/end) or recurring (rrule)
- **Config policies**: tag policies applied to all monitors in org
- **V1 vs V2**: Monitors core is V1 API, Downtimes and Config Policies are V2
